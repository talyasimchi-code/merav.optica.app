# מירב האופטיקה — אפליקציית תיאום תורים

יישום עבודה מלא (לא דמו) לתיאום תורים לבדיקת ראייה, לפי המסמך המקורי. Node.js +
Express בצד השרת, HTML/CSS/JS רגיל בצד הלקוח (בלי שלב build), ואחסון נתונים
בקובץ JSON מקומי (`data/store.json`) שנוצר אוטומטית בהרצה ראשונה.

## הרצה מקומית

```bash
npm install
cp .env.example .env
# ערכי את .env: שם משתמש/סיסמה לניהול, מספר הוואטסאפ של החנות, וכו'
npm start
```

- צד הלקוח: http://localhost:3000
- ממשק הניהול: http://localhost:3000/admin

## הגדרות חשובות ב-`.env`

| משתנה | מה זה |
|---|---|
| `ADMIN_USERNAME`, `ADMIN_PASSWORD` | פרטי ההתחברות לממשק הניהול |
| `SESSION_SECRET` | מחרוזת אקראית לחתימת עוגיית ההתחברות — **חובה לשנות** לפני עלייה לאוויר |
| `STORE_WHATSAPP` | מספר הוואטסאפ של החנות, בפורמט בינלאומי בלי + (למשל `97239696756`) |
| שעות הפתיחה, משך הבדיקה, רזולוציית הסלוטים | ניתנים לשינוי בלי לגעת בקוד |

## נקודה חשובה: סנכרון חגים (Hebcal)

`lib/hebcal.js` שולף את החגים מ-hebcal.com בזמן ריצה. **זה לא נבדק מול השרת החי**
כי הסביבה שבה נבנה הקוד הזה חסמה גישה לאותו דומיין (כל סביבת פיתוח מוגבלת
ככה מטעמי אבטחה). ברגע שהאתר עולה לאוויר על שרת רגיל עם גישה לאינטרנט, זה
אמור לעבוד — אבל **חשוב לבדוק את זה בפועל אחרי הפריסה הראשונה**: היכנסי לממשק
הניהול → "ניהול יומן" → "סנכרון חגים מלוח השנה העברי", ותוודאי שהחגים
הקרובים באמת מופיעים ברשימה.

אם משהו לא עובד שם, אפשר תמיד להוסיף חגים/ערבי חג באופן ידני דרך אותו מסך
("חסימה ידנית") עד שהבעיה תיפתר.

## פריסה לאוויר (hosting)

זה שרת Node.js סטנדרטי, אז כל שירות שמריץ Node עובד:

- **Render / Railway / Fly.io** — הכי פשוט: מחברים את הריפו, מגדירים את משתני
  ה-`.env` בממשק שלהם, ומריצים `npm start`.
- **VPS רגיל** — `npm install --production && npm start`, מומלץ מאחורי nginx
  עם HTTPS (Let's Encrypt) ומנוהל ע"י pm2 כדי שיקום אוטומטית אם קרס.

⚠️ **חשוב**: קובץ `data/store.json` הוא איפה שכל הלקוחות והתורים נשמרים. בפלטפורמות
כמו Render/Railway בתוכנית החינמית, מערכת הקבצים לפעמים *לא* נשמרת בין
פריסות (ephemeral filesystem) — כדאי לוודא עם הפלטפורמה שנבחרה שיש להן
"persistent disk"/"volume" ולחבר אותו לתיקיית `data/`, אחרת הנתונים יימחקו
בכל עדכון קוד.

## מה שלא נכלל (כמו במסמך המקורי)

- אפליקציה נייטיבית להורדה — זה אתר Mobile-first, נשלח כקישור.
- תשלום מקוון.
- שליחה אוטומטית מלאה של וואטסאפ (בכוונה — כל הודעה יוצאת מהחשבון האישי של
  בעלת החנות בלחיצת כפתור, בלי WhatsApp Business API).
- כמה אנשי צוות/חדרי בדיקה מקבילים.

## שדרוגים עתידיים אפשריים

- להחליף את `lib/db.js` במסד נתונים אמיתי (Postgres/SQLite) אם הנפח יגדל.
- להוסיף תזכורת אוטומטית (cron job) שבודקת תורים למחר ומכינה טיוטת הודעה,
  במקום ללחוץ ידנית בכל פעם.
